The Animator

The Animator is the execution layer of semantic rendering. It reads current state at display speed and applies it to Canvas, DOM, or another rendering target.

It runs sixty times a second. It doesn't know — or care — what the network is doing.

NOTE — In the original experiments (exp-01 through exp-11), this role was called the Painter and drew on Canvas2D. With A2UI integration, it becomes the Animator — same tween math, same 60fps loop, but driving CSS transforms on accessible HTML components instead of canvas pixels. The demo below still uses canvas to illustrate the concepts.
The Animator is a loop. Every ~16 ms it wakes up, reads what's on stage, samples every tween, and applies the current values to each component. It never waits. It never decides. It just animates.

First: the plain-English version

Imagine a security camera that takes a photo sixty times per second. It doesn't wait for something interesting to happen before snapping — it snaps constantly. If nothing changed since the last photo, the photos look identical. If a node is halfway through morphing, the photo captures it mid-morph.

That's the Animator. Every frame it asks: "What does the stage look like right now?" — and applies those values. The stage might have changed since last frame (a new patch arrived, a tween advanced) or it might be identical. The Animator doesn't know. It just animates.

The key insight: smooth animation isn't the Animator "scheduling" a move. It's the Animator repeatedly sampling a math function that says where something is at any given moment in time. Each frame is just a snapshot at a slightly later timestamp.

Three things happen every frame

step 1
◎

Read the stage

Loop over every node in the node map. Get its current position, size, kind, and any active tweens.

step 2
▲

Ask the tween

"Where are you right now?" The tween does the math — elapsed time, easing curve — and returns a number.

step 3
■

Render, then repeat

Clear the canvas, render every node at its current values, then schedule the next frame in ~16 ms.

Canvas implementation: clear and redraw every frame (immediate mode — the approach used in this demo). Retained component implementation: sample state and update transforms/properties on existing DOM or A2UI elements. The universal responsibility is the same: sample current state and apply it at display rate.

What is a tween, exactly?

A tween is not a timer or a scheduled event. It's a record with three numbers: from, to, and t0 (the timestamp when it started). That's it.

Every frame, the Animator asks: "Given that this tween started at t0 and lasts 1100 ms, and it's now time t — what fraction of the way through are we?" It plugs that fraction into an easing curve and gets back a number between from and to.

The math in one line:
value(t) = from + (to − from) × ease( (t − t₀) / duration )
This is a pure function. Call it at any time t and get the right value. No scheduler, no state machine, no events — just arithmetic.

Under the hood

The actual render loop — 8 lines of real code

This is the Animator (originally called the Painter in canvas experiments). It runs inside requestAnimationFrame, which the browser calls ~60 times per second, synchronized to your screen's refresh rate.

function frame() {
  ctx.clearRect(0, 0, W, H);          // wipe the canvas

  for (const node of stage.values()) {  // loop over every node on stage
    const x = cur(node, "x");           // ask the tween: "where is x right now?"
    const y = cur(node, "y");
    const a = alpha(node);              // enter / exit fade
    drawNode(node, x, y, a);            // render it
  }

  requestAnimationFrame(frame);         // schedule the next frame (~16 ms from now)
}

That's it. No conditions on whether data is arriving. No waiting. No checking the network. The loop wakes up, draws what it sees, and immediately schedules the next wake-up.

What cur(node, "x") does — the tween sampler

When a patch arrives (e.g. update card1 {x: 0.7}), the stage doesn't just set node.x = 0.7. Instead it stores a tween object:

node.tweens.x = {
  from: 0.2,    // where x was
  to:   0.7,    // where x should go
  t0:   now(),  // when this tween started
  dur:  1100    // milliseconds to complete
}

Then cur(node, "x") — called every frame — does this:

function cur(node, key) {
  const tw = node.tweens[key];
  if (!tw) return node[key];           // no tween → return the raw value

  const elapsed = now() - tw.t0;
  const u = clamp(elapsed / tw.dur);    // 0.0 → 1.0 progress
  const e = easeInOut(u);              // apply the easing curve

  if (u >= 1) {
    node[key] = tw.to;               // tween finished — clean it up
    delete node.tweens[key];
    return tw.to;
  }

  return tw.from + (tw.to - tw.from) * e; // interpolated value right now
}

The Animator calls this for every property of every node, every frame. Because it's just arithmetic — no timers, no events — it works at any frame rate and handles interruptions perfectly: if a new patch arrives mid-tween, you just replace tw.from with the current value and tw.t0 with now.

What is requestAnimationFrame?

requestAnimationFrame(fn) is a browser API that says: "Call fn before the next screen repaint." The browser syncs this to your display refresh rate — typically 60 Hz, sometimes 120 Hz.

Compared to…Difference
setInterval(fn, 16)rAF is synced to the actual screen; setInterval drifts and can fire when the tab is hidden (wastes battery). rAF pauses automatically when the tab is hidden.
A while(true) looprAF yields to the browser between frames, letting it handle input, layout, and other tasks. A while loop would freeze the page.
CSS animationsrAF gives you full control over what gets drawn. CSS animations are declarative; rAF is imperative — you can compute anything per frame.

The pattern function frame(){ … requestAnimationFrame(frame); } is a self-scheduling loop: each call to frame schedules the next one. It runs forever until you stop calling requestAnimationFrame.

Why redraw everything from scratch each frame?

This is called immediate-mode rendering (as opposed to retained-mode, where you keep a widget tree and update it). Canvas2D is immediate-mode: each frame you clear and repaint.

Why is this actually fine?

  • A modern device can clear and repaint a 900×500 canvas in well under 1 ms. Our scenes have ~5–20 nodes — this is trivial work.
  • There's no "diff" to compute. You don't need to figure out what changed — you just draw what's true right now.
  • Tweens and enter/exit animations fall out naturally: on every frame the values are slightly different, so the drawing is slightly different. No animation system needed beyond the tween sampler.
  • This is how games work (every frame, clear and redraw the whole scene). Dear ImGui, a widely used developer UI toolkit, works the same way.

The tradeoff: this approach doesn't scale to thousands of DOM-like elements. For Painted UI's use case (5–30 semantic objects per scene), it's the simplest possible thing that works.

What the Animator does NOT do
Not the Animator's jobWhose job it is
Deciding what nodes existThe Director (LLM)
Deciding which verb to use (flyin, dissolve, morph…)The Choreographer (motion policy)
Storing nodes between framesThe Stage (node map)
Handling network responsesThe parser loop (loop 3)
Knowing when a tween started or what caused itThe Stage — it stores the tween; the Animator just samples it

The Animator is the dumbest member of the cast on purpose. The dumber it is, the faster it runs, the harder it is to break, and the easier it is to A/B test with different visual styles.

Animator invariants:

  • Never waits for network input
  • Never calls the model
  • Never chooses semantic meaning
  • Does not own business state
  • Can be replaced without changing the semantic protocol
