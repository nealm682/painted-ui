The Cast — who does what in Painted UI

Five roles, one show. In the original experiments, the frame-rate loop drew on canvas and was called the "painter." With A2UI components, it drives CSS transforms on accessible HTML elements — so its role is now the Animator. Same tween math, same 60fps loop, different output target.
the LLM / agent
DirectorDecides WHAT exists and WHAT changes. Speaks only in semantic patches — never pixels, never geometry math. Expensive, slow (tokens), brilliant.
tokens over the wire
The ScriptThe patch stream — the director's instructions arriving a few characters at a time. Kilobytes, not megabytes.
motion policy layer
ChoreographerDecides HOW each change moves: verb, duration, easing, direction, stagger. A pure function, not a runtime agent. Can live in patch data (model decides, costs tokens — current exp-03) or as a local policy function (free, deterministic, versioned — the production target).
the frame-rate loop
AnimatorSamples tween state and applies CSS transforms to A2UI components at display rate. Owns the motion verb library. Never decides anything.
the node map (state)
StageOne shared data structure. Directors and choreographers WRITE it; the animator READS it. They never talk directly.
The economic logic of the whole cast: every decision moved down the chain moves from tokens (money, latency, variance) to code (free, instant, deterministic). The director is the only paid actor. Everyone else works for free, forever, on the user's own device.

Open the boxes — real objects, real streams

1 · A scene node — the atom of the stage

The stage (node map) holds nodes. A node is a small semantic object — note what it doesn't contain: no pixels, no animation timeline, no event handlers.

{
  "id": "card2",            // stable identity — this is what makes morphing possible:
                             // the same id later = "same thing, changed" → animate the difference
  "kind": "card",           // which painter routine draws it
  "x": 0.50, "y": 0.60,     // position as fractions of the viewport (resolution-independent)
  "w": 0.22, "h": 0.32,
  "img": 12,                // which cached asset fills it (a seed / cache key)
  "label": "Lighthouse",
  "z": 2,                   // paint order
  "semantic": "card: lighthouse day-trip",  // meaning — powers hit-testing, click messages
                             // back to the director, and (future) the accessibility mirror
  "interactive": true
}

The node is a noun. Nothing here says how it appears or moves — that's the choreographer's business. Separating the noun from the motion is the core trick.

2 · The script — what the director actually streams

One JSON object per line, arriving token by token. This is a real-shaped stream for a small scene (about 1 KB total — a 2 Mbps video stream sends this much every 4 milliseconds):

{"op":"palette","set":{"bg0":"#1a2438","accent":"#7fd4c1"}}   // set the mood first
{"op":"add","node":{"id":"bg","kind":"gradient","z":0}}        // back to front, like a painter
{"op":"add","node":{"id":"sun","kind":"disc","x":0.78,"y":0.22,"r":0.08,"glow":true,"z":1}}
{"op":"add","node":{"id":"title","kind":"text","x":0.5,"y":0.16,"text":"Weekend on the Coast","z":3}}
{"op":"add","node":{"id":"c1","kind":"card","x":0.22,"y":0.6,"w":0.22,"h":0.32,"img":5,"label":"Tidepools","z":2}}
{"op":"note","text":"laying in the stops…"}                   // optional narration (never a substitute!)
{"op":"update","id":"c1","set":{"x":0.5,"w":0.6,"h":0.5}}     // same id + new numbers = MORPH
{"op":"remove","id":"sun"}                                    // choreographer decides how it exits
{"op":"done"}

Four verbs of change — add, update, remove, palette — cover everything exp-01 through exp-09 do. exp-10 adds artistic verbs (strokeIn, materialize…) where the model names a behavior the painter already knows how to perform.

Key property: each line is useful the instant it closes. The interface assembles while the director is still talking — that overlap is the painted-live feeling.

3 · The choreographer — one patch in, one motion decision out

The choreographer is a pure function: (patch, scene, mood) → motion plan. It runs in microseconds, costs zero tokens, and is versionable like any design system. Here's an actual decision table (from the choreographer deep-dive) and a worked example:

function choreograph(patch, node, mood){
  if (patch.op === "add") {
    if (node.x < 0.22)  return {kind:"flyin",  from:"left",  dur:D(700)};  // geometry → direction
    if (node.x > 0.78)  return {kind:"flyin",  from:"right", dur:D(700)};
    if (node.kind === "text")           return {kind:"rise", dur:D(800)};  // kind → verb
    if (node.kind === "disc" && node.glow) return {kind:"zoom", dur:D(900)};
    return {kind:"dissolve", dur:D(800)};
  }
  if (patch.op === "update") return {tween:true, dur:D(1100), ease:"inout"}; // same id ⇒ morph
  if (patch.op === "remove") return exitTowardNearestEdge(node);
}
// D(ms) scales by mood: calm ×1.3 · neutral ×1.0 · energetic ×0.55
Patch arrivesChoreographer's decisionWhy
add card at x:0.22flyin from LEFT, 700 msposition near left edge → enter from that edge
add text "title"rise + settle, 800 mstext should land like set type, not fly
update c1 {w:0.6}tween w over 1100 mssame id = same thing changing → morph, never blink
remove sundissolve (center-ish node)exits mirror entries; ambience never snaps

Three possible homes for this decision: the model picks verbs itself (expressive, inconsistent, costs tokens), this local policy picks (consistent, free, testable), or hybrid — policy default, model override. Production wants B/C: the policy IS the brand's motion design system.

4 · The animator — sixty questions per second, zero decisions

The animator is a loop that runs at your display's refresh rate and does exactly this, every frame, forever — whether or not anything is streaming:

function frame(){
  for (node of stage)             // read the shared state — never waits for the network
    draw(node, valuesRightNow(node));  // ask each tween: "where are you at this instant?"
  requestAnimationFrame(frame);   // and again in ~16ms
}

valuesRightNow is the whole magic. A tween isn't a scheduled animation — it's a function of elapsed time:

x(t) = from + (to − from) × ease((t − t₀) / duration)   // closed-form: any frame rate, no error,
                                                        // no physics instability, no scheduler

The animator also owns the verb library — the motion implementations the director can name for one token each. With A2UI, verbs become CSS transform sequences on standard components. In the original canvas experiments, verbs like strokeIn drew directly — here's what that looked like:

// the director said only: {"verb":"strokeIn","id":"tower"}   ← ~8 tokens
// the painter expands that into:
path   = lighthouseOutline(anchors, viewport)   // ~40 points, parametric, any resolution
length = totalLength(path)
each frame: draw the first ease(t−t₀) × length of the path, tip-first,
            and let the brush cursor ride the current tip           // ← "drawn into existence"

Written once (~40 lines), reusable forever, swappable per theme. With A2UI components the verb library shifts from canvas drawing to CSS transform choreography — but the protocol and the tween math are identical.

5 · The stage — the only place the cast ever meets

The stage is one Map of id → node. The rule that makes four independent loops need no scheduler, no locks, and no coordination:

Director + Choreographer  →  WRITE the stage   (when patches arrive — bursty, unpredictable)
Animator                  →  READ the stage    (60×/sec, metronomic, never blocked)

Before and after one update patch:

// BEFORE                                    // AFTER {"op":"update","id":"c1","set":{"w":0.6}}
stage = {                                    stage = {
  c1: {kind:"card", w:0.22, …}                 c1: {kind:"card", w:0.22→0.6 over 1100ms, …}
}                                            }
// the node isn't replaced — it grows a tween. The animator just keeps asking
// "what's w right now?" and gets 0.22, 0.31, 0.44 … 0.60 across ~66 frames.

A node mounted mid-stream simply starts existing and animates in while later tokens are still arriving. Streaming mutates; rendering samples. The overlap is the show.

The one-paragraph summary to keep

The director (LLM) streams semantic patches — nouns and changes, never pixels. The choreographer (local policy) turns each change into a motion plan in microseconds. The stage (node map) absorbs the change as a tween. The animator (frame-rate loop) reads the stage sixty times a second and applies CSS transforms to A2UI components on the user's own silicon. Tokens buy meaning; code performs it; nothing waits for anything. The model directs; your device animates.
