Painted UI · an experiment in semantic rendering

How the Four Roles
Work Together

Painted UI streams semantic changes instead of pixels. Local policy turns those changes into governed motion, and a frame-rate executor performs them immediately on the user's device.

The model renders intent. The client renders experience.
Intent → Policy → Experience

The universal pattern

Semantic rendering is an architectural approach: the model expresses what should exist and what changed, not how to draw it. A local policy layer decides motion behavior. A frame-rate executor performs it.

Model intent → Semantic protocol → Local policy compiler → Shared state → Client experience

The theater metaphor below — Director, Script, Choreographer, Stage, Animator — is how Painted UI implements this pattern.

The Director interprets meaning. The Choreographer compiles behavior. The Stage mutates state. The Animator performs the experience.

Why four separate roles? The four clocks.

Each role runs at a completely different speed. If you chained them together, the slowest clock would freeze everything else. Decoupling them via shared state lets each run at its natural rate.

🤖 Conversation
1 per 5–30 s
📶 Network chunks
20–60 per s
🔍 Complete patches
unpredictable
🎨 Render frames
60 per s · steady

Chain these into one loop and the slowest clock wins. A patch arriving unpredictably would stall the render; the render running at 60fps would busy-loop the parser. Separation via shared state solves this cleanly.

Trace a single patch through the system

Step through what happens when the LLM sends {"op":"add","node":{...}}. Each step highlights the active role and shows what it receives and emits.

STEP 1 🤖 Director (LLM)
The model generates tokens. It doesn't send pixels or timelines — just a small JSON object describing a semantic change. This is one instruction in a stream of many.
Generates (token by token):
{"op":"add","node":{
  "id":"card1",
  "kind":"card",
  "x":0.18,"y":0.60,
  "w":0.28,"h":0.36,
  "label":"Q3 Growth",
  "semantic":"quarterly growth card"
}}
↓ Streams as raw text to the network
STEP 2 📶 Network Reader (Loop 2)
The browser's SSE reader receives chunks. Chunks arrive with no respect for JSON boundaries — one object might be split across three chunks, or three objects might arrive in one chunk.
Receives (3 chunks, boundary mid-object):
chunk 1: {"op":"add","node":{"id":"card
chunk 2: 1","kind":"card","x":0.18,"y":0
chunk 3: .60,"w":0.28,"h":0.36,"label":"Q3 Growth","semantic":"quarterly growth card"}}
↓ Appends each chunk to a text buffer
STEP 3 🔍 Parser / Assembler (Loop 3)
Scans the buffer character by character, tracking how many { vs } it has seen (brace depth). When depth returns to 0, one complete JSON object is ready. Only then does it fire.
Scans buffer, tracking brace depth:
depth=0  sees {  → depth=1  (object started)
depth=1  sees {  → depth=2  (nested object)
depth=2  sees }  → depth=1
depth=1  sees }  → depth=0  ← COMPLETE! Fire applyPatch()
↓ Calls applyPatch(patch) with the parsed object
STEP 4 🎵 Choreographer (Pure Function)
Not a loop. Called once per patch. Reads the patch's meaning — op type, node position, node kind — and returns a motion plan in microseconds. No tokens spent. No network call. Pure arithmetic.
Receives: {op:"add", node:{x:0.18, kind:"card", …}}
function choreograph(op, node, mood) {
  if (op === "add") {
    // node.x = 0.18, which is < 0.22
    // → it lives near the left edge
    // → it should enter FROM the left edge
    return {kind:"flyin", from:"left", dur:700};
  }
}
↓ Returns motion plan: {kind:"flyin", from:"left", dur:700}
STEP 5 🏳 Stage (Node Map)
The node is written into the shared Map. The motion plan becomes a tween — a record of where something starts, where it ends, when it started, and how long it takes. That's all a tween is.
applyPatch writes to the Map:
stage.set("card1", {
  id: "card1", kind: "card",
  x: 0.18, y: 0.60, w: 0.28, h: 0.36,
  label: "Q3 Growth",
  tweens: {
    x: {from:-0.20, to:0.18, t0:now(), dur:700}
    // starts off-screen left, slides to x=0.18
  },
  enterT: now()  // for alpha fade-in
})
↓ Node now exists in shared state — Animator will find it next frame
STEP 6 🎨 Animator (Loop 4 · 60 fps)
The Animator never stopped running. It doesn't know a patch arrived. It just wakes up every 16ms, reads the Stage, and asks each node: "where are you right now?" The tween answers with math.
Frame N+1 (16ms after patch): tween is ~2% done
// The Animator calls cur(node, "x") each frame:
elapsed = now() - t0        // e.g. 16ms
u = elapsed / dur           // 16/700 = 0.023
e = easeInOut(u)            // smooth curve ≈ 0.001
x = from + (to-from) × e   // -0.20 + (0.38 × 0.001) = -0.1996

// Frame N+22 (350ms in): tween is 50% done
u=0.5 → e=0.5 → x = -0.20 + 0.38×0.5 = -0.01

// Frame N+44 (700ms): tween complete
u=1.0 → x = 0.18  → tween deleted
↓ Draws the node at its current interpolated position. Every frame. Always.

Technical details

All 6 patch operations the Director can send

The entire vocabulary the LLM uses. Six operations. This is the protocol — the only thing that can cross the wire.

// 1. Set the color palette (mood, ambience)
{"op":"palette","set":{"bg0":"#1a2438","accent":"#7fd4c1"}}

// 2. Add a new node to the scene
{"op":"add","node":{"id":"card1","kind":"card","x":0.5,"y":0.6,"w":0.28,"h":0.35,"label":"Q3 Growth","semantic":"quarterly growth"}}

// 3. Update properties of an existing node (same id = morph, not replace)
{"op":"update","id":"card1","set":{"x":0.65,"w":0.45},"dur":1100}

// 4. Remove a node (choreographer decides exit animation)
{"op":"remove","id":"card1"}

// 5. Optional narration — never replaces a patch, just context
{"op":"note","text":"laying in the key cost driver…"}

// 6. Signal the stream is complete
{"op":"done"}

Node kinds: gradient, disc, rect, card, text, pill. Each kind is a drawing routine the Animator knows. The Director names a kind; the Animator executes it.

Why is the brace-depth parser necessary? (the boundary problem)

SSE delivers the model's output as a stream of arbitrary-sized chunks. There is no guarantee that a chunk aligns with any JSON boundary. Example:

// The model is generating this object:
{"op":"add","node":{"id":"sun","kind":"disc","x":0.78}}

// But it arrives in 3 chunks:
chunk 1: {"op":"add","nod
chunk 2: e":{"id":"sun","kind":"dis
chunk 3: c","x":0.78}}

You cannot call JSON.parse(chunk) — chunks 1 and 2 are invalid JSON. The parser accumulates chunks into a buffer and scans for complete objects by tracking brace nesting depth:

scan each character:
  '{' → depth++
  '}' → depth--
  depth === 0 → complete object! → JSON.parse() → applyPatch()

This is why you need Loop 3 as a separate concern from Loop 2. The network reader just appends bytes. The parser is the one who understands JSON structure.

What is a tween? Why not use CSS transitions?

A tween is a record stored on a node:

node.tweens.x = {
  from: 0.20,     // starting value
  to:   0.65,     // target value
  t0:   1721450001234,  // timestamp when tween started (performance.now())
  dur:  1100      // milliseconds to complete
}

The Animator samples it each frame with pure math:

elapsed = performance.now() - t0
u = clamp(elapsed / dur, 0, 1)    // fraction: 0.0 → 1.0
e = easeInOut(u)                  // smooth S-curve
value = from + (to - from) * e    // interpolated result

Why not CSS transitions?

What the Stage (node map) holds at any moment

At any point mid-stream, the Stage might look like this — some nodes fully settled, one mid-tween, one entering:

stage = Map {
  "bg"    → { kind:"gradient", settled:true, tweens:{} },

  "sun"   → { kind:"disc", x:0.78, y:0.22, r:0.09,
               tweens: {},           // fully settled
               enterT: 1721450001000  },

  "card1" → { kind:"card", x:0.18, y:0.60, w:0.28,
               tweens: {
                 x: {from:-0.20, to:0.18, t0:1721450001800, dur:700}
                 // ← mid-animation right now
               }},

  "title" → { kind:"text", x:0.50, y:0.15,
               tweens: {
                 y: {from:0.22, to:0.15, t0:1721450001900, dur:800}
               }}
}

The Animator reads this map every 16ms. For nodes with tweens, it samples the math. For nodes without tweens, it uses the raw values directly. There is no scheduler, no animation queue, no event system.

Design constraints of this architecture

These are the invariants that keep the roles cleanly separated. Violating any one collapses the separation of concerns the architecture depends on.

1
The Director never renders frames.
2
The Animator never makes semantic decisions.
3
Rendering never waits for generation.
4
The rendering target is replaceable.
Six more constraints
  1. Only allow-listed semantic instructions cross the model boundary.
  2. Stable identity represents continuity — a node's id is its persistent self.
  3. Policy is versioned separately from content.
  4. The same patch stream is loggable and replayable.
  5. Accessibility is derived from the same semantics, not bolted on afterward.
  6. Reduced motion changes policy, not meaning.

What breaks when responsibilities collapse?

Each invariant exists because its absence produces a predictable failure.

Collapsed responsibility Result
Model emits pixels or frame instructions High bandwidth, high latency, poor accessibility
Model chooses every routine transition Token waste, visual variance, weak brand governance
Network arrival controls rendering Motion stalls when the stream stalls
Nodes lack stable identity Updates become remove-and-add; continuity disappears
Renderer owns business meaning Rendering target becomes hard to replace
