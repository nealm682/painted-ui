The Choreographer

The layer that decides how things move — and why it's the difference between a demo and something a large company can actually ship.

The theater metaphor (the whole architecture in one breath)

The Directorthe LLM/agent. Decides what exists and what changes: "add three cards; focus the second; retitle the scene." Speaks in semantic patches.
The ChoreographerA design concern, not a runtime agent. Decides how each change moves: verb, duration, easing, stagger. Implemented as a pure function (~40 lines) or delegated to the model. Three architectures: model-picks (A, current exp-03), local-policy-picks (B), hybrid (C, production target).
The Painterthe compositor (exp-01/02/03). Executes the motion 60×/s on the user's own device. Never makes decisions.
The Stagethe node map — the shared state all four loops meet at.
Separating these is what makes the system scale: directors are expensive, choreographers and painters are free. Every decision you move down the chain moves from tokens (money, latency, variance) to code (free, instant, deterministic).

The policy itself — motion from semantics

A choreographer is ~40 lines of rules that read each patch's meaning and output motion parameters. This is the actual decision table used in the demo below:
function choreograph(op, node, scene, mood){
  // WHERE does it enter from? Geometry is semantics.
  if(op==="add"){
    if(node.x < 0.22)        return {kind:"flyin", from:"left",  dur:D(700)};
    if(node.x > 0.78)        return {kind:"flyin", from:"right", dur:D(700)};
    if(node.kind==="text")   return {kind:"rise",  dur:D(800)};   // text settles like set type
    if(node.kind==="disc"&&node.glow) return {kind:"zoom", dur:D(900)}; // focal points bloom
    return {kind:"dissolve", dur:D(800)};                          // default: paint in
  }
  if(op==="update")  return {tween:true, dur:D(1100), ease:"inout"}; // same id ⇒ FLIP morph, never remove+add
  if(op==="remove"){
    if(node.x < 0.35) return {kind:"flyout", to:"left",  dur:D(600)};
    if(node.x > 0.65) return {kind:"flyout", to:"right", dur:D(600)};
    return {kind:"dissolve", dur:D(550)};
  }
  if(op==="palette") return {tween:true, dur:D(1400)};              // ambience never snaps
  }
  // mood scales everything: D(ms) = ms × (calm 1.3 | neutral 1.0 | energetic 0.55)
  // stagger: each mount waits (calm 260 | energetic 110) ms after the previous — the painter's rhythm
Note what the rules key on: position → direction, node kind → verb, id-match → morph, mood → tempo. Nothing here needs a model. This is why it can run in 0ms on every patch — including patches that interrupt other patches.

Scaling this in a large enterprise (vendor-neutral)

The choreographer is not a competitor to declarative UI protocols and agent frameworks — it's the feel layer that sits on top of exactly that kind of stack:
LayerEnterprise realityWhere painted-UI fits
Agent runtimeserver-side agent frameworks (e.g., AWS Strands: model-driven loop, tools, multi-agent, observability — used in production at AWS itself)unchanged. The agent is the Director; it just streams patches instead of prose.
UI protocoldeclarative, allow-listed formats (e.g., Google's A2UI: flat component list, ID references, built to stream, no arbitrary code execution)near-perfect substrate. A2UI's ID-referenced incremental updates ≈ our patch ops. Add a transition-hint vocabulary + client choreographer and A2UI messages start to feel painted.
Motionusually: whatever CSS shippedthe choreographer = a motion design system as versioned code — brand-reviewed once, enforced everywhere, A/B-testable, no model variance in front of customers.
Governance / securityno arbitrary code from models; audit trailsdeclarative patches are loggable and replayable; the policy is reviewable code; constrained decoding can make invalid patches unemittable.
Accessibilitynon-negotiableevery node carries semantic — the same field that drives hit-testing can drive screen-reader mapping; prefers-reduced-motion is just another mood.
Scale profilemillions of users, burstyKB-sized payloads = ordinary web traffic; rendering cost is distributed to client devices; only real per-interaction cost is tokens — reducible via choreographer (B), caching, and on-device SLMs.
The honest enterprise summary: the parts that scale hard (GPU, rendering, motion) cost ~nothing in this architecture; the part that costs (tokens) is per-interaction, shrinkable, and already the cost every agent product accepts. The risk isn't economics — it's craft: whether the policy's taste survives contact with a design-system committee.
