Why Four Loops?

The architecture behind the live painter (exp-03) — and why no single loop can fake a screen being painted by a model.
The one-sentence version: streaming mutates state, rendering samples state, and four processes at four different speeds only ever meet in one place — the node map.

The problem: four things happen at four speeds

~5–30 sa conversation turn (you ask, you click) ~20–60 ×/snetwork chunks arrive (SSE deltas) per chunkJSON objects complete (unpredictably) 60 ×/sthe screen must repaint, no matter what
These clocks don't divide into each other. Chain them into one loop and the slowest clock wins — the screen freezes at network speed. That's a loading page, not a painting.

What goes wrong with fewer loops

ArchitectureWhat the user sees
1 loop — wait for full reply, then drawspinner… then a finished page. Zero illusion.
2 loops — draw when data arrivescontent pops in frozen steps; motion dies between chunks; a stalled network stalls the art.
3 loops — stream + render, naive line parserour run 1 & 2 failures: chunks split mid-object, ops silently lost.
4 loops — decoupled via shared stateevery fragment mounts the instant it completes, animations run through network stalls, ambient motion never stops.

The four loops, one job each

1 · Conversation
the director's contract
One model call per interaction. Clicks are messages, so the model mutates its own scene instead of repainting.
2 · Network read
the courier
await reader.read() — delivers arbitrary fragments. Makes no promises about boundaries.
3 · Parser scan
the assembler
Brace-depth scan over a buffer. Emits exactly one applyPatch() per completed object. Absorbs all boundary chaos.
4 · Render
the animator
requestAnimationFrame, 60×/s. Reads the node map, applies tween values as CSS transforms to A2UI components. Never waits for anything.

The meeting point: one shared state

loops 1–3 write the node map  ·  loop 4 reads it
no queues, no timeline, no coordination — a node mounted mid-stream simply starts existing, and its enter animation is a function of time-since-arrival
This decoupling is the illusion: structure lands at token speed while motion runs at frame speed. The overlap between "still arriving" and "already animating" is what reads as being painted live.

Where this pattern comes from (honest lineage)

  • Game engines — the canonical decoupled loop: logic at fixed timestep, rendering as fast as the hardware allows. Nystrom's Game Programming Patterns "Game Loop" chapter; Fiedler's "Fix Your Timestep." Unity/Unreal/Godot all work this way.
  • Streaming protocols — SSE/chat streaming taught everyone that partial output should be shown immediately; producer/consumer buffering is decades old.
  • Incremental parsers — tokenizers that maintain state across arbitrary chunk boundaries are standard compiler/network practice.
  • Immediate-mode rendering — redraw everything every frame from current state (Canvas, Dear ImGui), instead of mutating a widget tree.
None of the four loops is novel. The composition is the design: wiring token-speed streaming to a frame-rate compositor — with a motion policy layer between them — to reproduce the painted-live feel without video generation.

The open gap we're working in: generative UI exists and streams; nobody in this list is adding transition choreography on top of a standard component protocol to reproduce the video-model feel. That synthesis — and the four-role decomposition it tests — is the wiki's original contribution.

"But I don't have a GPU"

You already ran it. That's the point.
  • Canvas2D is rasterized by the browser using whatever the machine has — integrated graphics or plain CPU compositing targets our whole motion vocabulary at 60fps (unverified on low-end hardware; formal traces are a TODO). "The user's GPU" was shorthand for the user's own silicon, whatever it is.
  • The economic claim never depended on the client having a good GPU — it depends on the server not needing one at all. Flipbook rents an H100-class stream per viewer; we ship 1–2 KB of structure and let a 10-year-old laptop do the rest.

Small models on-device: yes — three roles

  • The choreographer. A small local model picks transition verbs and layout while a big cloud model authors content. Tiny structured task, perfect for a 1–4B model.
  • The whole painter, offline. WebLLM runs Llama/Phi/Gemma/Qwen-class models in the browser via WebGPU at up to ~80% native speed, OpenAI-compatible API, streaming + JSON mode included. Zero marginal cost, zero network latency, private.
  • The reliability fix — this one matters most. Local inference allows grammar-constrained decoding (llama.cpp GBNF, Outlines): invalid tokens are masked at generation time, so patches are guaranteed valid against our protocol grammar. Runs 1–2 (narration, .5 decimals) become structurally impossible — the model literally cannot emit them.
SLM caveat: composition taste may drop before syntax does — that's an empirical question for exp-05.

How to explain it to someone in 30 seconds

"Products like Flipbook make UI feel alive by generating every pixel with a video model — magical, but you're renting a GPU per viewer.

We get the same feeling by having the model choreograph with instructions instead of pixels — a few kilobytes of semantic patches, streamed live, animated on your own device. The model is the director; your machine is the animator; the token stream is the choreography score."
